# 01 · Getting Started

> **Audience:** any developer joining HAAT NOW. After this page you can run the platform locally, log in to
> every portal, and understand what you are looking at.

## Purpose
Get a working local environment in minutes and orient you to the four surfaces (customer, driver, merchant,
admin) that all ship from a single React SPA.

## Architecture (the 30-second version)
HAAT NOW is **one** React 19 + Vite 6 + TypeScript SPA. It runs in one of two backend modes, chosen entirely by
the `VITE_AUTH_MODE` env var:

- **`sandbox`** (default, and what production ships as) — the backend is `localStorage`. No network, no database
  required. Demo accounts + seeded data. This is the demo everyone sees.
- **`supabase`** — real Supabase (Postgres + Auth + Storage + Realtime). Opt in with `HAAT_LIVE_BACKEND=1`.

The same components, services, theme engine, and RBAC run in both modes. Mode only swaps the storage layer at the
bottom. See [18-multi-tenancy.md](18-multi-tenancy.md) and [17-authentication.md](17-authentication.md).

## Flow: run it locally
```bash
npm install
npm run dev            # Vite dev server on http://localhost:3000
```
That's it — no `.env` needed for sandbox mode (the default). `GEMINI_API_KEY` in `.env.local` is only used by
optional AI features.

**Log in** (any portal, same login screen — role is derived from the phone number):

| Portal | Phone | OTP |
|---|---|---|

Demo accounts + OTP are defined in [`src/services/auth.service.ts`](../../src/services/auth.service.ts)
(`DEMO_ACCOUNTS`, `SANDBOX_OTP`).

## Flow: the essential commands
| Command | What it does |
|---|---|
| `npm run dev` | Dev server (port 3000) |
| `npm run lint` | **Typecheck** — `tsc --noEmit`. Must be **0 errors**. |
| `npm run build` | `vite build` + stamps `dist/version.json` (git SHA) |
| `npm run preview` | Serve the production build locally |
| `node docs/testing/e2e_runner.cjs` | Puppeteer E2E suite (target **24/24**) |

## Dependencies
- **Node.js** (the only prerequisite for sandbox mode).
- React 19, Vite 6, TypeScript, Tailwind v4 (via `@theme` in `src/index.css`), i18next, `@supabase/supabase-js`.
- For live mode only: a Supabase project + env keys.

## Extension points
- New surface/route → `src/App.tsx` (role router). New portal feature → `src/features/<surface>/`.
- New env-driven behavior → gate on `import.meta.env.VITE_AUTH_MODE` (never on `import.meta.env.DEV`; see the
  warning in `auth.service.ts`).

## Reuse rules
- **Never** talk to `localStorage` or Supabase from a component — go through a service (see
  [04-service-architecture.md](04-service-architecture.md)).
- **Never** hardcode a color/radius/font — read a design token (see [06-theme-engine.md](06-theme-engine.md)).

## Files involved
- [`src/main.tsx`](../../src/main.tsx) — provider tree / boot order.
- [`src/App.tsx`](../../src/App.tsx) — role router (customer/driver/merchant/admin).
- [`src/services/auth.service.ts`](../../src/services/auth.service.ts) — demo accounts, OTP.
- [`vite.config.ts`](../../vite.config.ts) — forces sandbox in production unless `HAAT_LIVE_BACKEND=1`.
- [`package.json`](../../package.json) — scripts.

## Do's
- ✅ Run `npm run lint` before every commit — it is the CI gate.
- ✅ Use the demo accounts above; they have valid UUID ids so Supabase-mode queries never `22P02`.
- ✅ Read [02-repository-structure.md](02-repository-structure.md) next.

## Don'ts
- ❌ Don't add an `&& import.meta.env.DEV` gate to auth mode — it breaks login on the deployed build.
- ❌ Don't commit real Supabase keys. Sandbox mode needs none.
- ❌ Don't edit `dist/` — it is generated by `npm run build`.

## Example: confirm your setup
```bash
npm install && npm run lint      # → 0 errors
npm run dev                      # open http://localhost:3000
# log in as +201000000005 / 123456 → you land in the Admin Dashboard
```
